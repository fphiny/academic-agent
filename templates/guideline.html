전체 화면 가이드

컬렉션 · 문서관리 · 웹문서 페이지 가이드라인

첨부된 실제 화면 이미지를 기준으로, 각 페이지의 목적 · 탭 구조 · 사용자 흐름 · API 역할 · 상태 표현 방식을 한 번에 정리한 운영/개발 가이드입니다. 화면 구조를 유지하면서 기능 수정, API 연동, QA 점검을 할 때 기준 문서로 사용할 수 있습니다.

공통 UI 패턴

• 좌측 사이드바는 모든 화면에서 동일한 정보 구조를 사용한다.
• 상단 헤더는 페이지 제목 + 서브타이틀 + 상태 배지 + 우측 사용자 버튼으로 구성된다.
• 본문 상단에는 Hero 영역이 있고, 화면 목적과 요약 지표를 함께 보여준다.
• 주요 기능은 탭 또는 단계형 탭으로 전환되며, 활성 탭은 파란색으로 강조된다.
• 폼 화면은 좌측 입력/작업, 우측 API·상태 보조 패널의 2단 구조를 기본으로 한다.
• 결과 탭은 작업 로그 + Raw API 결과를 함께 보여주는 패턴을 재사용한다.

개발 시 핵심 원칙

• 탭 전환은 화면 이동이 아니라 동일 페이지 내 패널 교체로 처리한다.
• 컬렉션 목록은 생성/삭제 후 자동 새로고침되어야 한다.
• 문서 화면은 컬렉션 목록을 공통 데이터 소스로 사용한다.
• 웹문서는 1 → 2 → 3 → 4 단계 플로우가 깨지지 않도록 상태를 공유해야 한다.
• 모든 작업은 사용자가 확인할 수 있도록 로그와 Raw API Result에 누적되어야 한다.
• 입력 미리보기(JSON, 메타데이터, 선택된 URL, 선택 요소)는 사용자 실수를 줄이기 위한 필수 UX다.

페이지별 목적 요약

컬렉션 관리

지식베이스의 컬렉션을 생성하고, 목록 조회·메타데이터 편집·count 조회·삭제·alias 관리를 수행하는 관리자 화면이다.

문서 관리

텍스트/파일 업로드부터 검색, 전체 조회, 수정, 삭제까지 문서 라이프사이클을 한 화면에서 관리하는 화면이다.

웹문서 처리

URL을 입력해 요소를 선택하고, 링크를 추출한 뒤, 선택된 URL을 컬렉션으로 임베딩하는 단계형 워크플로 화면이다.

컬렉션 관리 가이드

생성 / 목록 / Alias / 유틸리티 / 결과 탭의 역할과 구현 포인트를 설명한다.

컬렉션 생성

신규 컬렉션 이름, 도메인, 설명을 입력하고 metadata preview를 확인한 뒤 생성한다.

화면 기준

설명 포인트

  • 컬렉션 이름과 도메인은 텍스트 입력 필드로 받는다.
  • metadata.engine은 고정값 cosine으로 유지된다.
  • 설명 textarea와 JSON 미리보기는 실시간 동기화되는 것이 좋다.
  • 생성 완료 후 컬렉션 목록과 Alias select 데이터가 갱신되어야 한다.

관련 엔드포인트 / 상태

생성 APIPOST /api/db/collections
목록 갱신자동 refresh
Alias select자동 sync

구현 메모

미리보기 JSON은 사용자의 입력이 API payload에 어떻게 반영되는지 보여주는 UX 장치다.
우측 패널은 실제 API 바디 구조를 명시해 운영자와 개발자가 같은 규격을 보게 한다.

컬렉션 목록

카드형 레이아웃으로 컬렉션 메타데이터를 보여주고 count / edit / delete 작업을 수행한다.

화면 기준

설명 포인트

  • 컬렉션명, 설명, engine, domain, name을 한 카드에 묶어 표시한다.
  • 상세 메타데이터는 접기/펼치기 구조로 보조 정보에 적합하다.
  • metadata edit, count, delete 액션은 카드 단위로 실행된다.
  • 목록 새로고침 버튼은 서버 상태를 강제로 다시 읽는 액션이다.

관련 엔드포인트 / 상태

목록 조회GET /api/db/collections
Count 조회GET /api/db/collections/:name/count
삭제DELETE /api/db/collections/:name

구현 메모

카드형 목록은 컬렉션 수가 적당히 많을 때 가독성이 좋다.
삭제 후 빈 목록이 되더라도 empty state가 자연스럽게 노출되어야 한다.

Alias 관리

컬렉션 별 별칭(alias)을 저장하고, 별칭 삭제와 현재 alias 목록 확인을 수행한다.

화면 기준

설명 포인트

  • 좌측은 저장 영역, 우측 상단은 삭제, 우측 하단은 live 목록 확인 영역으로 분리된다.
  • 대상 컬렉션 select는 현재 컬렉션 목록과 항상 동기화되어야 한다.
  • Alias는 여러 별칭을 쉼표 기준으로 입력받는 구조로 확장 가능하다.
  • 현재 Alias 목록에는 collection → alias 관계가 그룹 형태로 표시된다.

관련 엔드포인트 / 상태

Alias 저장POST /api/aliases/set
Alias 삭제DELETE /api/aliases/delete
Alias 목록GET /api/aliases

구현 메모

Alias 삭제는 오타 복구나 명명 정책 정리에 자주 사용될 수 있으므로 빠른 입력 UX가 좋다.
live 배지는 현재 목록이 서버 데이터와 동기화된다는 인상을 준다.

유틸리티

선택 컬렉션의 count 조회와 컬렉션 삭제를 분리해 보수적으로 처리하는 보조 탭이다.

화면 기준

설명 포인트

  • 좌측 count 조회, 우측 삭제라는 위험도 차이에 맞춰 시각적 구분이 있다.
  • 삭제 영역은 붉은 톤 테두리와 버튼으로 위험 액션임을 강조한다.
  • 삭제 안내 문구는 목록/선택 박스 자동 갱신 규칙까지 설명해야 한다.

관련 엔드포인트 / 상태

CountGET /api/db/collections/:name/count
DeleteDELETE /api/db/collections/:name

구현 메모

운영자 실수 방지를 위해 삭제 전 confirm 단계 또는 2차 확인을 추가할 수 있다.
유틸리티 탭은 상세 관리보다 단일 목적 액션에 집중하도록 단순 유지하는 것이 좋다.

작업 결과

컬렉션과 Alias 관련 작업 로그와 마지막 Raw API 응답을 함께 누적 표시한다.

화면 기준

설명 포인트

  • 좌측 로그는 사람이 읽는 작업 이력, 우측은 디버깅용 JSON 응답이다.
  • 로그 비우기/Raw 비우기 버튼은 테스트 환경에서 반복 확인에 유용하다.
  • 실패·성공 메시지를 모두 남겨 재현성을 높이는 것이 중요하다.

관련 엔드포인트 / 상태

로그 대상생성/목록/Alias/삭제
Raw 응답마지막 API Result

구현 메모

운영자 관점에서는 로그, 개발자 관점에서는 Raw JSON이 중요하므로 두 패널이 모두 필요하다.
결과 탭은 다른 탭에서 실행된 액션도 누적 반영되어야 한다.

문서 관리 가이드

텍스트 삽입 / 파일 업로드 / 검색 / 조회 / 삭제 / 결과 탭의 흐름을 기준으로 설명한다.

텍스트 문서 삽입

본문 텍스트를 직접 입력하여 chunking + embedding + DB 저장을 수행한다.

화면 기준

설명 포인트

  • Collection, Doc ID, 제목, 카테고리, 출처, 본문 텍스트를 입력한다.
  • Metadata 미리보기는 title / source / category 구조를 보여준다.
  • Doc ID를 비우면 자동 생성되며, 사용자가 입력 시 강제 id로 처리된다.
  • 우측 패널은 ingest 엔드포인트와 입력 파라미터를 설명한다.

관련 엔드포인트 / 상태

엔드포인트POST /api/db/documents/ingest
필수 파라미터collection, text
메타데이터title / source / category

구현 메모

본문이 길어질 경우 스크롤 처리와 textarea 높이 정책이 중요하다.
컬렉션 select는 반드시 공통 컬렉션 목록 API와 연결되어야 한다.

파일 업로드 삽입

여러 파일을 업로드하고 서버에서 순차 저장하는 FORM-DATA 기반 탭이다.

화면 기준

설명 포인트

  • Collection, Doc ID Prefix, 카테고리, 출처, Files 입력을 받는다.
  • 파일 선택 수는 Hero 요약 카드와 동기화된다.
  • 메타데이터 미리보기는 업로드 공통 값(category, source)을 보여준다.
  • 선택된 파일이 없을 때는 우측 상태 카드에 empty state를 표시한다.

관련 엔드포인트 / 상태

엔드포인트POST /api/db/documents/upload
전송 형식multipart/form-data
파일 파라미터files[]

구현 메모

다중 업로드에서는 진행 상태 또는 업로드 결과 목록이 있으면 운영성이 좋아진다.
대용량 파일이 포함되면 비동기 업로드 또는 chunk 업로드 고려가 필요하다.

문서 검색

벡터 유사도 검색으로 관련 문서를 찾는 검색 탭이다.

화면 기준

설명 포인트

  • Collection, Top K, Where(JSON), Query 입력을 받는다.
  • 검색 결과는 Raw JSON 또는 별도 결과 리스트 형태로 표시할 수 있다.
  • Where(JSON)은 메타데이터 필터로 사용되며 JSON 형식 오류 검증이 필요하다.
  • Top K는 기본값을 두되 숫자 검증이 필요하다.

관련 엔드포인트 / 상태

엔드포인트GET /api/db/search
필수collection, query
옵션k, where

구현 메모

검색 버튼 이후 결과가 없을 때도 명확한 empty result 메시지가 필요하다.
Where 예시 placeholder는 사용자의 JSON 작성 부담을 줄여준다.

전체 문서 조회

컬렉션 전체 문서를 받아와 테이블로 표시하고, 클라이언트 측 필터와 수정 기능을 제공한다.

화면 기준

설명 포인트

  • Batch Size와 Start Offset으로 조회 범위를 제어한다.
  • Where(JSON) 필터와 테이블 필터는 역할이 다르다. 서버 조건 vs 클라이언트 필터.
  • 상단 배지로 GET /api/db/documents, POST /api/db/documents/upsert, JS Filter를 명시한다.
  • 테이블 행 단위 편집은 수정 모달 또는 inline edit 방식으로 확장될 수 있다.

관련 엔드포인트 / 상태

조회GET /api/db/documents
수정POST /api/db/documents/upsert
필터JS client filter

구현 메모

조회 탭은 데이터량이 많아질 가능성이 높으므로 pagination 또는 virtual scroll을 검토할 수 있다.
클라이언트 필터는 이미 받은 rows에만 적용된다는 점을 안내 문구로 분명히 보여준다.

문서 삭제

삭제 대상 문서를 먼저 조회한 뒤, 선택 삭제 또는 행 삭제를 수행하는 위험 액션 탭이다.

화면 기준

설명 포인트

  • 조회 탭과 유사한 구조지만 붉은 강조색으로 삭제 맥락을 명확히 한다.
  • 삭제 대상 전체 조회 후 선택 삭제, 필터 초기화 흐름을 제공한다.
  • 삭제용 Where(JSON)과 테이블 필터를 함께 제공해 정확한 대상 선택을 돕는다.
  • 행 삭제와 선택 삭제 구분은 실수 방지에 중요하다.

관련 엔드포인트 / 상태

엔드포인트DELETE /api/db/documents
보조 액션row delete / selected delete

구현 메모

삭제는 조회 단계 없이 바로 실행하지 않도록 현재 구조를 유지하는 것이 안전하다.
실행 후 로그와 Raw API 응답에서 삭제 건수/대상을 확인 가능해야 한다.

작업 결과

문서 삽입/업로드/검색/조회/삭제 작업 로그와 Raw API 응답을 누적 관리한다.

화면 기준

설명 포인트

  • 문서 관련 모든 탭의 액션이 이 결과 탭으로 모인다.
  • 작업 로그는 사용자 친화적인 텍스트, Raw API Result는 개발용 원문이다.
  • 비우기 버튼은 테스트 시나리오 반복에 유용하다.

관련 엔드포인트 / 상태

로그 범위ingest/upload/search/read/delete
Raw 응답마지막 API Result

구현 메모

문서 화면은 기능이 많아 결과 탭의 중요도가 특히 높다.
가능하면 성공/실패 badge나 timestamp를 함께 표기하면 운영성이 더 좋아진다.

웹문서 처리 가이드

1. URL 입력 / GPT 추천 → 2. URL 추출 → 3. 임베딩 추출 → 4. 결과 로그 단계로 구성된다.

1단계 — URL 입력 / iframe / GPT 추천

페이지 URL을 입력하고 iframe 미리보기, 요소 선택, GPT 추천을 함께 처리하는 시작 단계다.

화면 기준

설명 포인트

  • 페이지 불러오기, GPT 추천, 추천 요소 강조, 선택 초기화, 추출 단계 이동 버튼이 있다.
  • iframe 미리보기는 hover/선택/추천 강조 상태를 시각적으로 다르게 표현한다.
  • 선택 요소 영역은 마지막 클릭 요소 1개만 유지한다.
  • 추천 결과는 추천 ID / 추천 CLASS 텍스트영역으로 관리된다.

관련 엔드포인트 / 상태

카카오톡 채널 RAG 연동 가이드

카카오톡 채널에서 들어온 사용자 발화를 /api/rag/kakao로 전달하고, 서버가 RAG 답변을 카카오 응답 포맷으로 반환하는 연동 방식입니다. 현재 코드 기준으로 즉시 응답 방식과 callbackUrl 비동기 응답 방식을 모두 지원합니다.

Skill Endpoint
/api/rag/kakao
POST JSON
응답 포맷
simpleText
version 2.0
Callback 지원
useCallback
긴 RAG 답변용

1. 전체 연동 흐름

1
카카오톡 채널에서 사용자 메시지 입력

사용자가 채널 채팅방에서 질문을 입력하면 카카오 스킬 서버로 JSON payload가 전달됩니다.

2
FastAPI가 발화와 callbackUrl 추출

서버는 userRequest.utterance, utterance, value.resolved, value.origin, message 순서로 사용자 메시지를 읽습니다. callbackUrl은 userRequest.callbackUrl, callbackUrl, action.clientExtra.callbackUrl 순서로 확인합니다.

3
RAG 답변 생성

answer_query_stream()을 내부적으로 실행하지만, 카카오 응답에서는 streaming이 아니라 최종 답변 텍스트만 모아서 반환합니다.

4
카카오 simpleText 응답

callbackUrl이 없으면 즉시 template.outputs.simpleText.text로 답변하고, callbackUrl이 있으면 먼저 useCallback: true를 반환한 뒤 백그라운드 스레드가 callbackUrl로 최종 답변을 전송합니다.

2. 카카오 스킬 서버 설정값

MethodPOST
Content-Typeapplication/json
Endpointhttps://도메인/api/rag/kakao
ResponseKakao v2 simpleText
카카오 서버에서 접근 가능해야 하므로 로컬 주소가 아니라 외부 HTTPS 도메인 또는 터널링 URL을 사용해야 합니다.

3. 서버 동작 기준

로그인 필수 아님

카카오 route는 require_login()을 호출하지 않습니다. 로그인 세션이 있으면 로그를 저장하고, 없으면 RAG 답변만 수행합니다.

마크다운 금지 프롬프트 자동 추가

사용자 질문 뒤에 “일반 텍스트로만 답변” 지시가 붙습니다. 카카오 채팅 UI에서 깨지는 마크다운을 막기 위한 처리입니다.

Alias 해석

전달된 collection은 내부에서 alias를 실제 collection_name으로 해석합니다.

Fallback 메시지

질문이 없으면 “사용자 메시지를 읽지 못했습니다.”, 오류가 나면 “지금 답변이 어려워요. 잠시 후 다시 시도해 주세요.”를 반환합니다.

4. 요청 Payload 예시

카카오 기본 payload 외에 collection, model, k, where 등을 함께 넘길 수 있습니다.

{
  "userRequest": {
    "utterance": "장학금 신청 방법 알려줘",
    "callbackUrl": "https://callback-url-from-kakao"
  },
  "collection": "scholarship",
  "model": "gemma3",
  "mode": "normal",
  "think": false,
  "k": 5,
  "where": "{\"category\":\"장학금\"}",
  "sid": "kakao-user-session-id",
  "history_limit": 20
}
where 주의: 현재 route는 parse_json_str()로 처리하므로 JSON 객체를 문자열로 넘겨도 됩니다.

5. 응답 포맷 예시

즉시 응답
{
  "version": "2.0",
  "template": {
    "outputs": [
      {
        "simpleText": {
          "text": "장학금 신청 방법은 ..."
        }
      }
    ]
  }
}
callbackUrl이 있을 때 1차 응답
{
  "version": "2.0",
  "useCallback": true
}
callbackUrl이 있으면 서버는 백그라운드 thread를 띄우고, 최종 답변을 callbackUrl로 다시 POST합니다. callback POST timeout은 15초입니다.

6. 테스트 절차

서버 단독 테스트

curl 또는 Postman으로 /api/rag/kakao에 JSON을 보내 simpleText 응답이 오는지 확인합니다.

카카오 스킬 테스트

카카오 스킬 서버 URL에 외부 HTTPS endpoint를 입력하고 발화 테스트를 실행합니다.

실채널 테스트

채널 채팅방에서 질문 후 답변 속도, callback 응답, 로그 저장, collection alias 해석을 확인합니다.

curl -X POST "https://도메인/api/rag/kakao" \
  -H "Content-Type: application/json" \
  -d '{
    "userRequest": {"utterance": "입학 장학금 알려줘"},
    "collection": "scholarship",
    "k": 5,
    "history_limit": 20
  }'
